    Scriptable Objects

    Contenus de la page

    4-4 Classes scriptables

    Temps requis

    45 minutes

    Les MonoBehaviour qui sont amplement utilisés dans les précédents exemples gèrent le comportement d'un GameObject dans le jeu. Ces objets ont un comprotement plus " classique " en contenant des données et des comportements comme une classe standard, de même que plusieurs instances qui sont créées en cours d'exécution.

    Les ScriptableObject permettent de définir un type de Asset particulier basé sur un script donné. Ils sont structurés par une classe qui hérite de ScriptableObject et chaque instance est un asset différent. Leur intégration dans le jeu ressemble plus à celle d'une énumération dans un projet standard.

    Créer un ScriptableObject

    Un ScriptableObject est une classe qui hérite de ScriptableObject. Il faut lui ajouter l'attribut de classe CreateAssetMenu qui ajoute dans le menu contextuel des assets (clic droit dans la fenêtre Assets) une option pour créer des instance du ScriptableObject. L'attribut de classe comprend trois propriétés.

    Propriété Signification Type de valeur Obligatoire
    fileName Le nom du fichier d'asset créé par défaut Chaîne de caractères Oui
    menuName Le nom du menu où l'option se situe (des sous-menus peuvent être créés avec la barre oblique). Chaîne de caractères Oui
    order L'ordre d'apparition dans le menu Entier Non

    Les méthodes des ScriptablesObjects

    Les méthodes suivantes des ScriptableObjects peuvent être appelés par la boucle de jeu.

    • Awake : à la création du ScritpableObject;
    • OnDestroy : à la destruction du ScriptableObject;
    • OnEnable / OnDisable : lorsque l'objet est activé ou désactivé.

    Les autres méthodes de la boucle de jeu ne sont pas appelés sur les ScriptableObjects, car il ne s'agit pas d'objets dans le jeu. De plus, la méthode CreateInstance permet de créer dynamiquement de nouveaux ScriptableObjects alors que Instantiate permet de dupliquer un ScriptableObjects (comme pour les GameObject).

    Chargement et utilisation

    On peut charger les ScriptableObject avec une référence comme pour les Prefab et les GameObject de la hiérarchie. On peut aussi récupérer la liste des instances d'un ScriptableObject d'un certain type avec le système des resources d'Unity.

    Pour fonctionner, les instances doivent être situées dans un dossier appelé Resources (avec ou sans R majuscule). Le dossier n'est pas obligé d'être à la racine des assets, il peut contenir des sous-dossiers et il peut y avoir plusieurs dossiers Resources dans un même projet.

    Chargement dynamique

    La classe Resources peut être utilisée avec n'importe quel type d'asset (image, son, texture, materiel... ), sauf pour les prefabs.

    Comment utiliser les ScriptableObject

    On utilise les ScriptableObject dans plusieurs contexte, dont trois principaux : regroupement de données communes et définition de type de données, gérer un comportement dynamiquement, partager des données entre des objets (singleton).

    Alléger les données en regroupant les données communes

    Prenons l'exemple d'un jeu de course dans lequel le joueur peut choisir différentes options pour sa voiture, comme une finition. Chaque finition change la couleur de la voiture, la teinte des vitres et la présence ou non d'enjoliveur de roues. Il serait inutile de répliquer plusieurs fois chaque GameObject de voiture pour tenir compte des variations (et imaginez le nombre à faire s'il y avait aussi des options de moteurs, des extras...).

    Donc chaque GameObject de voiture aura un fini qui lui est propre et indépendent du modèle (on peut peindre n'importe quelle voiture en jaune par exemple). Le ScriptableObject qui gère cela se définit comme suit.

    Gérer des comportements dynamiques

    Les ScriptableObject peuvent aussi être utilisées pour gérer les comportements des objets. Revenons sur l'exemple du jeu de course qui permet d'ajouter un bonus spécial à chaque voiture (boost de vitesse, invicibilité temporaire...). Chacune de effets a une durée et un temps de récupération.

    public abstract class Voiture : MonoBehaviour
{
    [SerializeField]
    private float vitesse;
    public Vitesse 
    {
        get {return vitesse;} 
        set {vitesse = value;}
    }

    [SerializeField]
    private float estInvicible;
    public EstInvicible 
    {
        get {return estInvicible;}
        set {estInvicible = value;}
    }

    [SerializeField]
    private Bonus bonus;

    private bool peutUtiliserBonus;

    private void Awake()
    {
        peutUtiliserBonus = true;
    }

    public void FixedUpdate() 
    {
        rigibody.position += vitesse * direction * Time.deltaTime;
    }

    public void OnCollisonEnter() 
    {
        if (EstInvicible) 
        {
            return;
        }
    }

    public void UtiliserBonus(InputAction.CallbackContext contexte)
    {
        if(peutUtiliserBonus)
        {
            StartCoroutine(AppliquerBonus);
        }
    }

    private IEnumerator AppliquerBonus()
    {
        peutUtiliserBonus = false;
        bonus.AppliquerEffet(voiture);
        yield return new WaitForSeconds(bonus.Duree);
        bonus.RetirerEffet(voiture);
        yield return new WaitForSeconds(bonus.TempsRecuperation);
        peutUtiliserBonus = true;
    }
}

    Machine à état et ScriptableObject

    On pourrait voir les états d'une machine à états comme étant des ScriptableObject, mais comme le code diffère entre chaque état, ceux-ci seraient sûrement utilisés qu'une seule fois, ce qui rend inutilit la création de ScriptableObject.
