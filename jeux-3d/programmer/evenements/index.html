    Programmation événementielle

    Contenus de la page

    3-4 Patron de conception observateur
    3-5 Programmation événementielle

    Temps requis

    30 minutes

    En programmation séquentielle, les méthodes et les instructions s'exécutent les unes à la suite des autres dans une séquence explicite et définie. Mis à part quelques cas, dont les indirections dues au polymorphisme ou l'utilisation de la réflexivité, on peut toujours prédire la prochaine instruction exécutée à la simple lecture du code source. Voici un exemple de traitement séquentiel permettant de réagir au clic d'un bouton.

    // Bouton générique 
public abstract class Bouton {   
    public abstract void Cliquer();
}

// Spécialisation pour incrémenter le compteur
public class BoutonIncrement : Bouton
{
    public Compteur Compteur { get; set; }

    public override void Cliquer() {
        Compteur.Incrementer();
    }
}

// Spécialisation pour décrémenter le compteur
public class BoutonDecrementer : Bouton
{
    public Compteur Compteur { get; set; }

    public override void Cliquer() {
        Compteur.Decrementer();
    }
}

// Gère le compteur d'un entier
public class Compteur {
    public int Valeur { get; private set; }

    public Compteur () {
        Valeur = 0;
    }

    public void Incrementer() {
        Valeur ++;
    }

    public void Decrementer() {
        Valeur --;
    }
}

    Lorsqu'on clique sur le bouton dans l'interface, la méthode abstraite Cliquer du type bouton est appelée (sur l'instance BoutonIncrementer ou BoutonDecrementer selon le cas).

    Le principal défaut de cette approche est le nombre de classes nécessaires. Il serait bien plus simple d'avoir un bouton générique qui permet de gérer toutes les actions. L'idée est de fournir au bouton une méthode a exécuter lorsqu'il est cliqué. Par contre, on ne peut pas le faire via une interface, car chaque objet géré (comme le Compteur) aurait un seul traitement disponible. On peut alors utiliser une idée semblable aux pointeurs de fonctions de C, des foncteurs C++ ou des interfaces fonctionnelles de java (Bref, chaque langage à dû inventer un concept pour adresser ce problème).

    Types délégués

    En C# un type délégué est simplement la signature d'une méthode que l'on utilise comme type de variable. On peut assigner aux variables du type délégué n'importe quelle méthode respectant la signature définie par le type délégué. La syntaxe d'un délégué est :

    Déclaration d'un type délégué
    1
    [public | protected | private | internal] delegate TYPE_RETOUR NOM_DELEGUE ([parametre1[, parametres2...]]);

    Par exemple le délégué suivant répond à la signature de la méthode A et B, mais pas à celle de la méthode C. On peut s'en servir comme type pour stocker de l'information.

    Le modificateur de visibilité ici intervient au même titre que celui devant une variable, une méthode ou une classe. Donc même si la méthode B est privée et que MonDelegue est public, la méthode B répond aux critères de MonDelegue.

    Dans notre exemple de bouton, on va donc définir un délégué dans Bouton pour spécifier que Bouton peut envoyer un message à toutes les méthodes sans paramètres. (Notez le Del à la fin du nom du type délégué, c'est une nomenclature standard en C#). L'idée est qu'au lieu de créer des spécialisations du comportement du bouton à chaque action différente que celui-ci doit accomplir, le comportement (méthode à exécuter) sera une donnée du bouton.

    Événements

    À présent, on doit trouver une manière d'informer Bouton quelle méthode doit être appelé lorsqu'il est cliqué. On va utiliser le concept d'événements pour ce faire. Un événement permet de lancer un traitement à un moment précis en brisant le flux d'exécution. L'avantage à utiliser des événements est qu'il n'est pas nécessaire de savoir quel objet doit recevoir l'information; c'est de la responsabilité des objets qui veulent être notifiés de s'abonner à l'événement.

    Une petite analogie ici : suivre un abonnement c'est un peu comme suivre quelqu'un sur Facebook, c'est à vous de vous abonner et non la personne qui va vous forcer à recevoir l'information. L'approche présentée au début correspond plus au Publi-sac où vous recevez une tonne d'information que vous n'avez pas demandée et qu'ils vont porter chez vous (ils savent qui vous êtes 😦).

    Une événement C# correspond à un type délégué que l'on assigne à une « variable » spécial qui gère les événements. La syntaxe d'un événement est la suivante.

    Déclaration d'un événement
    1
    [public | protected | private | internal] event TYPE_DELEGUE NOM_EVENEMENT;

    Encore une fois, les modificateurs de visibilités servent seulement à définir la visibilité de l'événement par rapport à sa classe. Dans bouton, on peut maintenant ajouter un événement à exécuter lorsque le bouton est cliqué. On retrouve souvent le mot « On » pour identifier un événement.

    Déclaration de l'événement pour répondre au clic de la souris
    1
2
3
4
    public class Bouton {
    public delegate void ActionClicDel ();
    public event ActionClicDel OnClicBouton;
}

    Pour exécuter un événement, on utilise la méthode !#c# Invoke sur l'événement. Il faut toutefois s'assurer que celui-ci n'est pas nul (qu'il y a au moins un objet pour le recevoir à l'autre bout...). L'opérateur de vérification de nullité !#c# ? permet de réaliser cela simplement.

    // Bouton générique 
public class Bouton {   

    public delegate void ActionClicDel ();
    public event ActionClicDel OnClicBouton;

    public void Cliquer() {
        OnClicBouton?.Invoke();
    }
}

    Remarquez ici que !#c# Bouton est maintenant un type concret et qu'il n'y a plus d'utilité à maintenir les classes !#c# BoutonIncrement et !#c# BoutonDecrement. Il ne reste plus qu'à ajouter !#c# Compteur pour qu'il écoute le bouton.

    Notez que l'opérateur !#c# += permet d'ajouter un abonné à un événement (donc la méthode indiquée sera appelée) et !#c# -= permet de se désabonner de l'événement.

    Désabonnement

    Il est important de toujours se désabonner quand on abonne à une référence d'objet, car cela peut causer des erreur de référence nulle. Un exemple de cette situation est donné plus-bas.

    On modifie donc le !#c# Bouton pour qu'il reçoive à sa création l'action qu'il devra faire et il abonne l'événement du clic à l'action à accomplir.

    // Bouton générique 
public class Bouton {   

    public delegate void ActionClicDel ();
    public event ActionClicDel OnClicBouton;

    // Référence vers le delegate utilisé
    private ActionClicDel actionBouton;

    public Bouton(ActionClicDel actionClic) {
        actionBouton = actionClic;
        OnClicBouton += actionClic;
    }

    public void Cliquer() {
        OnClicBouton?.Invoke();
    }
}

Donc nous ne ferons pas la suppression (nous verrons un autre exemple plus tard où nous pouvons le gérer).

    Types délégués de System

    L'espace de nom System propose deux types délégués génériques Action et Func. Les deux acceptent des paramètres génériques. Pour Action, tous les paramètres génériques sont des paramètres des méthodes acceptés. Pour Func, le premier est un type de retour, tous les autres sont des paramètres des méthodes acceptés. 

    public delegate void Action();
public delegate void Action<T1>(T1 param1);
...

public delegate TRes Func<TRes>();
public delegate TRes Func<TRes, T1> (T1 param1);
...

    Les utiliser peut simplifier le développement de l'application, car il n'est plus nécessaire de définir le délégué. Par exemple, on peut réécrire notre classe Bouton comme suit, en remplaçant notre délégué personnalisé par Action.

    UnityEvent

    Unity définit un délégué appelé UnityEvent avec des déclinaisons qui acceptent jusqu'à 4 paramètres (semblable à Action). On abonne un objet avec AddListener (plutôt que +=) et on le désabonne avec RemoveListener (plutôt que -=). On le déclenche aissoavec la méthode Invoke. Aucun UnityEventne retourne de valeur. De plus, il est possible de les affecter directement dans l'éditeur.

    Dans l'utilisation des UnityEvent, on peut lier des objets dont la signature ne correspond pas à celle du délégué UnityEvent dans les conditions suivantes :

    • La méthode n'accepte aucun paramètre, ce qui cause les valeurs de l'événements à être ignorées.
    • La méthode accepte que des paramètres natifs à c#, alors on doit indiquer la valeur passée à la méthode. Cette valeur sera constante à chaque appel de l'événement.

    dynamic

    Lorsque l'événement accepte des paramètres, les méthodes qui acceptent exactement le même de paramètres seront listés dans la section dynamic. Cela passera dynamiquement la valeur indiqué lors de l'invocation à la méthode gestionnaire de l'événement.

    Forces et faiblesses

    La force de la programmation événementielle est qu'elle permet une plus grande modularité dans la programmation et réduit les dépendances dans le code. Également, on peut partager une information à plusieurs abonnés sans effort d'implémentation supplémentaire.

    La grande faiblesse de la programmation événementielle est le débogage, car contrairement à la programmation séquentielle, la programmation événementielle ne permet pas de suivre le flux d'exécution (ordre d'exécution des méthodes et instructions) simplement en lisant le code source. Par conséquent, suivre le flux d'exécution pour traquer une erreur peut se révéler difficile, surtout sans débogueur spécialisé.

    On utilise donc la programmation événementielle lorsque la gestion des dépendances pose un problème. On l'utilise souvent pour remplacer l'anti-pattern Singleton.
