    On remarque les éléments suivants dans la création du formulaire :

    1. Chaque champs est accompagné d'un Label qui ne contient souvent pas de fx:id car il n'est pas nécessaire d'y faire référence dans le contrôleur.
    2. Chaque champs qui peut généré un message d'erreur est accompagné d'un Label qui est vide, mais contient un fx:id car il est nécessaire d'y faire référence dans le contrôleur pour mettre à jour les erreurs.
    3. La propriété promptText permet d'afficher un texte par défaut dans un contrôle de type texte.
    4. Les ComboBoxseront remplies du côté du formulaire. On peut le faire aussi dans le FXML, mais cela allourdit considérablement le code, surtout si la liste est longue.
    5. Le contrôle DatePickerpermet de gérer l'affichage d'un calendrier pour sélectionner la bonne date.
    6. Les contrôles RadioButton possèdent tous un toggleGroup pour les lier et indiquer qu'un seul peut être sélectionné. Le ToggleGroup est créé avec les lignes suivantes dans le FXML (juste après la création du VBox).

    Populer les ComboBox

    En JavaFX, il ne faut pas manipuler le constructeur lorsqu'on veut accéder à la logique d'initialisation. Il faut plutôt déclarer une méthode appelée initialize avec l'annotation @FXML. Cette méthode est automatiquement appelée par JavaFX lors de la création du contrôleur.

    Pour les ComboBox elles doivent contenir une ObservableArrayList qui se comporte exactement comme une ArrayList standard (avec les mêmes méthodes, sauf pour la création) et permet à l'interface de se mettre à jour lorsqu'on change les éléments de la liste (une liste qui implémente le patron observateur).

    @FXML
    private ComboBox<String> choixPaysDepart;

    @FXML
    private ComboBox<String> choixPaysArrive;

    @FXML
    public void initialize()
    {
        ObservableList<String> listePays = FXCollections.observableArrayList("Canada", "États-Unis", "France");
        choixPaysDepart.setItems(listePays);
        choixPaysArrive.setItems(listePays);
    }
}

    À la ligne 12, on aurait pu passer des objets dans un tableaux ou une ArrayLst à la méthode observableArrayList de FXCollections qui agit comme une Fabrique d'ObservableArrayList.

    Implémenter la validation

    Pour respecter les bons principes d'affectation de responsabilité (une méthode, une tâche), chaque type de validation aura une méthode dédiée. On va structurer la validation pour pouvoir réutiliser le même code pour valider la ville et le pays de départ ou d'arrivée.

    Validation des boîtes de sélection

    Le code ci-dessous s'assure qu'une valeur ait été sélectionné dans la ComboBox indiquée en paramètre. Si l'item sélectionné est nul, alors le Label voit sa valeur modifiée pour prendre le message d'erreur. Dans le cas contraire, on met le contenu du Label à vide.

    La mtéhode getSelectionModel retourne un objet de type SelectionModel qui est utilisé pour gérer la sélection dans les contrôles de sélection. Il offre plusieurs méthodes pour la sélection unique et multiple. On appelle getSelectedItem qui nous retourne un String car la ComboBox contient des Stirng.

    private boolean validerPays(ComboBox<String> choixPays, Label messageErreur) {
    boolean valide = choixPays.getSelectionModel().getSelectedItem() != null;

    // Message d'erreur
    String message = valide ? "" : "Un pays doit être sélectionné";
    messageErreur.setText(message);

    return valide;
}

    Rendre la validation dynamique

    Pour rendre la validation dynamique, on utilise la propriété focus pour déterminer lorsqu'un objet permet le focus. Les propriétés en java, qui sont développées dans la bibliothèque JavaFX, sont en fait des valeurs observables. Donc sur la propriété focused on ajoute un observateur qui implémente l'interface ChangeListener :

    Signature d'un observateur de changement en java
    1
2
3
4
5
6
    @FunctionalInterface
public interface ChangeListener<T> {
    void changed(ObservableValue<? extends T> observable,
            T oldValue,
            T newValue)
}

    La méthode changed reçoit la propriété observée qui a déclencheé la mise à jour, l'ancienne valeur et la valeur après le changement. Il s'agit d'une interface fonctionnelle, soit une interface qui possède une seule méthode. Plutôt que de toujours déclarer ses types dans un classe à part entière (fichier distinct) ou dans une classe anonyme, on peut simplement les utiliser comme des expressions lambdas.

    Donc dans notre exemple, on veut, lors de la perte de focus, déclencher la méthode de validation du pays. On attache donc le ChangeListener suivant à la propriété focused de choixPaysDepart, le ComboBox à observer.

    // Dans la méthode initialize
choixPaysDepart.focusedProperty().addListener((observable, oldValue, newValue) -> {
    if(!newValue) {
        validerPays(choixPaysDepart, erreurPaysDepart);
    }
});

    Nom des paramètres

    Le nom des paramètres de la lambda est arbitraire, seule leur position importe. On pourrait très bien les renommer en français.

    Sélection du mode de transport

    Aucune liaison vers la propriété focused a été faite pour les boutons radios, car leur fonctionnement assure qu'ils sont cochés lors de la perte de focus.

    Calculer le total du voyage

    Pour calculer le coût total du voyage, notre formulaire va devoir réagir au changement de valeurs dans les champs et non simplement à la perte de focus. On peut ajouter notre code de calcul à l'observateur de focused pour les champs qui ont déjà un observateur de ce type. Par contre, pour les boutons radios, les cases à cocher et le contrôle d'entier, on pourra réagir au changement de la valeur plutôt qu'à la perte de focus ce qui se traduit par une interface plus réactive.
