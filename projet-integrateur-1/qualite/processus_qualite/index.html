    Tests et débogage

    Les tests sont l'exécution du système pour identifier des zones qui contiennent une erreur. On peut faire des divers types de tests pour rechercher des problèmes pour divers critères de qualité logicielle. Le débogage est l'exécution du système pour identifier la nature des erreurs et la façon de les régler. Pour effectuer un débogage efficace il faut d'abord localiser les erreurs et entre en mesure de les reproduire. Il s'agit de deux activités complémentaires.

    Dans la vidéo ci-dessus, le test constitue l'action de la docteure et le débogage serait de recherche les connexions nerveuses (pour le moins inusitées) qui font bouger les oursons.

    Les tests logiciels

    Ils constituent une grande partie du processus d'assurance qualité. Il s'agit d'une activité de la catégorie « Validation et vérification (V&V) ».

    • On les utilise pour vérifier le fonctionnement adéquat du système.

    • Qu'est-ce que les tests prouvent ?

    • Si un scénario donné provoque une erreur / état indésirable dans le système.

    • Qu'est-ce que les tests ne prouvent pas ?

    • Que le système fonctionne adéquatement.

    Types de test

    • Tests d'interface utilisateur
    • Tests unitaires
    • Tests d'intégration
    • Tests de régression
    • Tests de charge
    • Tests d'acceptation
    • Vérification
    • Validation
    • "Monkey testing"

    Tests d'interface utilisateur

    But : Valider l'utilisabilité du système

    • Quoi tester ?
    • Les interfaces et les contrôles
    • La navigation
    • Comment le test est-il effectué ?
    • Inspection par des experts (voir grille heuristique)
    • Mise en situation des utilisateurs
    • Questions et entrevues pour les utilisateurs

    Exemples de critères heuristiques

    • Adéquation à la tâche (fonctionnalité appropriée)
    • Contrôles utilisateurs (degré de contrôle, contrôles explicites)
    • Flexibilité (possibilité de personnalisation, d'adaptation)
    • Gestion des erreurs (tolérance, prévention et correction)
    • Compatibilité (conformes aux attentes des utilisateurs, naturelle et intuitive)
    • Autodescription (rétroaction, guidage et support)
    • Consistence (emplacements, format, syntaxe et nommage)
    • Charge de travail (mémorisation minimale, concision et charge mentale)

    Voir document : UI Évaluation heuristique

    Tests unitaires

    But : Tester une unité du programme

    • Quoi tester ?
      • Une fonctionnalité publique
      • Limiter chaque cas de test à une méthode
    • Comment le test est-il effectué ?
      • Écrire un bout de code qui appelle la méthode à tester
      • Créer au besoin des stubs / driver

    Tests unitaires

    Déterminez 3 cas de tests pertinents pour la fonction suivante. Indiquez les entrées et la sortie attendue.

    public static int Somme(int a, int b) {
  return a + b;
}

    Tests unitaires



    Scénario Entrée "a" Entrée "b" Sortie attendue Sortie réelle Conclusion
    1 5 10 15 15 Valide
    2 -18 -12 -30 -30 Valide
    3 2033258858 114254789 2147513647 -2147453649 Fautif

    Que faire avec le scénario 3 ?

    • Lancer une exception de débordement (overflow)
    • Indiquer dans la documentation que la fonction n'est pas sécuritaire pour les débordements
    • Changer la structure d'entier pour utiliser de larges nombres
    • Tout cela est guidé par l'analyse de cas d'exécution plausibles.

    Tests d'intégration

    But : Tester la combinaison de plusieurs fonctionnalités

    • Quoi tester ?
    • Une fonctionnalité publique de haut niveau
    • Limiter chaque cas de test a une fonctionnalité
    • Comment le test est-il effectué ?
    • Écrire un bout de code qui appelle la méthode à tester
    • Valider le résultat que plusieurs parties du système ont produit

    Tests unitaires et d'intégration

    Limites :

    • Le test comporte du code, donc si l'on fait une erreur dans le code du test, on peut chercher une erreur qui n'existe pas ou en introduire dans le système.
    • C'est long et fastidieux à écrire.

    Tests de régression

    But : Vérifier que les nouvelles fonctionnalités n'introduisent pas de bogues dans les fonctionnalités existantes

    • Quoi tester ?
    • Tout le système potentiellement impacté par les changements
    • Comment le test est-il effectué
    • Effectuer les tests unitaires et les tests d'intégration déjà utilisés

    Tests de régression

    ## Utilité * Permet d'éviter la dégradation du système lors de l'ajout de fonctionnalités * Permet une détection des effets de bord à plus grande échelle et des effets de bord inattendu.
    ## Limites * Devient rapidement très lourd, car beaucoup de tests sont effectués. * La sélection ou priorisation conduit certaines parties à ne pas être testées (ou insuffisamment testées), donc des erreurs peuvent se glisser.

    Tests de charge (stress test)

    But : vérifier que le système supporte la charge de travail qui lui est demandée (nombre d'utilisateurs, concurrence des opérations ou gestion de la mémoire).

    • Quoi tester ?
    • Scénarios réels d'exécution (nombre de clients)
    • Comment le test est-il effectué ?
    • Simulation d'utilisateurs
    • Utilisation sans arrêt (fuites de mémoire)

    Tests de charge

    ## Utilité * Permet de limiter les risques au déploiement et à l'augmentation de la charge de travail * Permet d'identifier les limites d'utilisation du système
    ## Limites * Très difficile d'identifier les causes des erreurs signalées (quel buffer devient plein, est-ce que c'est le matériel qui lâche...) * Demande beaucoup de ressources à mettre en place

    Tests d'acceptation - Vérification

    But : Vérifier si le système satisfait aux spécifications établies par le client et qu'il est prêt à être déployé.

    • Quoi tester ?
    • Tester l'ensemble des fonctionnalités du système en cas réel. (Évidemment, l'ensemble des scénarios plausibles ne peuvent pas être testés.)
    • Comment le test est-il effectué ?
    • Utilisation du système par des utilisateurs cobayes

    Une fois les tests d'acceptation complétés, le système est considéré "release-candidate"

    Tests d'acceptation - Validation


    But : Vérifier si le système satisfait aux spécifications établies par le client et qu'il est prêt à être déployé.

    • Quoi tester ?
    • Réaction du client face au système
    • Comment le test est-il effectué ?
    • Rencontre avec le client et discussion sur les entrants et sortants du système.
    • Quand le test d'acceptation est-il fait ?
    • Comme il s'agit du seul test pour les spécifications, à chaque rencontre client, une période devrait être utilisée pour confirmer le respect des spécifications.

    Une fois les tests d'acceptation complétés, le système est considéré "release-candidate"

    Monkey testing

    But : Tester le système en utilisant des entrées aléatoires / quelconques / carrément erratique / qui visent à faire saigner du nez les développeurs afin de tester la robustesse du système.

    • Quoi tester ?
    • N'importe quel champ / mécanisme permettant d'envoyer des données à l'application
    • Comment tester ?
    • Utiliser des outils de génération de données aléatoires
    • Le faire dans l'application ou par test unitaire

    Monkey testing

    ## Utilité * Permet de valider la sécurité de l'application * Permet de faire surgir des erreurs rares ou que l'on peut difficilement prévoir (exemple accent ExpoSAT)
    ## Limites * Ne reflète pas l'utilisation réelle du système (honnêtement qui entrerait XÆA-Ⅻ dans le champ nom ?) * Peut prendre beaucoup de temps pour la valeur retirée (ex. toutes les fonctionnalités passent par la même fonction de validation des entrées.)

    Faire un plan de test

    1. Définir les tests d'acceptation
    2. Séparer les critères du test d'acceptation en catégories (unitaire, intégration, charge)
    3. Définir les entrées des tests et résultats attendus
    4. Tests unitaires
    5. Tests d'intégration
    6. Tests de charge

    Mettre en place un plan de test

    De façon régulière : 1. Choisir un outil de gestion des tests unitaires / d'intégration 2. Exécuter tests unitaires et intégrations à étape importante (on revient là-dessus bientôt) 3. Exécuter au besoin les tests de régression pertinents

    Avant un déploiement: 1. Exécuter les tests charges / monkey testing 2. Exécuter les tests d'acceptation

    Que faut-il tester ?

    • Éléments de code critiques :
    • Complexes
    • Fortement dépendants
    • Longs
    • Qui ont beaucoup d'enfants
    • Souvent utilisés
